## 결론 먼저

이 글의 결론은 다음과 같다.

- **무엇이 잘 됐는가**: 문서를 먼저 만들어 두니 AI가 IA·URL·콘텐츠 모델을 임의로 결정하지 않았다.
- **무엇이 아쉬웠는가**: 문서가 너무 많아지면 오히려 AI가 모순된 지시를 만든다. 7개 정도가 한계였다.
- **누구에게 추천하는가**: AI에게 코드뿐 아니라 "어떤 제품을 만들지"까지 맡기려는 사람.
- **다시 쓴다면**: 첫날부터 `CLAUDE.md`에 "하지 말 것"을 명시한다.

## 왜 이걸 해봤나

블로그를 바이브코딩으로 만들기 시작하면 LLM이 자기 판단으로 URL 구조, 카테고리, frontmatter 필드를
바꾸기 시작한다. 한 번 시작된 자유 결정은 다음 세션에서도 누적되지 않는다. 매번 다른 답이 돌아온다.

그래서 코드를 짜기 전에 "사람이 결정한 것"과 "AI가 결정해도 되는 것"을 분리하기로 했다.

## 사용 환경

- 사용 도구: Claude Code (CLI), Cursor (Astro 코드 편집)
- 운영체제: Windows 11
- 비용: 도구 구독 외 추가 비용 없음

## 목표

1. AI가 "임의 결정"하면 안 되는 항목을 모두 문서로 못 박기
2. 문서를 7~10개 이내로 유지하기
3. CLAUDE.md 한 장으로 LLM이 무엇을 따라야 하는지 알게 하기

## 진행 과정

### 1단계 — 결정 항목 추출

먼저 LLM이 자유롭게 했을 때 매번 답이 갈리던 항목을 적었다.

- 카테고리 이름과 slug
- URL 구조 (`/post/slug` vs `/posts/slug` 같은 사소한 차이)
- frontmatter 필드 (특히 `tags`, `series`, `tools`)
- 시리즈 개념의 유무
- AdSense 슬롯 위치
- About / Privacy 페이지의 톤

### 2단계 — Source of Truth 매핑

각 결정마다 "이 결정은 어떤 문서에 적혀 있는가"를 정했다. 한 결정이 여러 문서에 걸쳐 있으면 LLM은
어느 쪽을 따라야 할지 모른다.

| 주제 | 문서 |
|---|---|
| 카테고리/슬러그 | `CONTENT_MODEL.md` |
| URL 구조 | `SERVICE_IA.md` |
| 글 frontmatter | `CONTENT_MODEL.md` |
| 광고 정책 | `SEO_ADSENSE_CHECKLIST.md` |
| 기술 스택 | `ARCHITECTURE.md` + `ADR-001` |

### 3단계 — CLAUDE.md에 "금지" 섹션을 넣기

LLM은 "하지 말 것" 목록에 더 잘 반응한다. 예를 들어:

```text
- 클라이언트 SPA로 글 본문 렌더링하지 않는다.
- 모든 글에 같은 meta description을 쓰지 않는다.
- 빈 카테고리/태그 페이지를 대량 생성하지 않는다.
```

긍정 명령("정적 HTML로 만들어라")보다 부정 명령("SPA로 만들지 마라")이 같은 의도를 더 좁게 표현한다.

## 사용한 프롬프트

작업 시작 시 매번 같은 프롬프트로 컨텍스트를 고정했다.

```text
먼저 CLAUDE.md를 읽고, 그 다음 docs/README.md에 적힌 순서대로 문서를 읽어라.
문서와 충돌하는 URL 구조나 콘텐츠 모델을 임의로 만들지 마라.
필요한 변경이 있으면 먼저 어떤 문서를 수정해야 하는지 제안한 뒤 구현하라.
```

마지막 한 줄이 핵심이다. "필요하면 문서부터 고쳐라"라고 적어두면 LLM이 슬그머니 다른 결정을 끼워 넣지 않는다.

## 결과

- **성공한 부분**: 새 세션을 시작해도 AI가 같은 URL 구조와 frontmatter를 만든다.
- **예상과 달랐던 부분**: 문서를 너무 자세히 적으면 LLM이 그 안에서 모순을 찾아낸다. 짧게 유지하는 편이 낫다.
- **걸린 시간**: 문서 7개 작성에 반나절.

## 실패한 부분

- 처음에 `OPEN_QUESTIONS.md`에 적힌 질문을 LLM이 임의로 답해 버린 적이 있다.
  → 해결: `CLAUDE.md`에 "OPEN_QUESTIONS의 항목을 임의 결정하지 않는다" 한 줄을 추가.
- ADR(Architecture Decision Record)을 빠뜨린 결정이 있었다.
  → 해결: 큰 기술 결정은 무조건 ADR 한 장을 추가하기로 규칙화.

## 사람이 수정한 부분

LLM이 만든 카테고리 이름이 영어와 한국어가 섞여 일관성이 없었다. 사람이 일괄로 영어 슬러그 + 한국어
표시 이름 형태로 바꿨다.

## 최종 평가

| 항목 | 평가 |
|---|---|
| 재현성 | 같은 프롬프트로 같은 산출이 나옴 |
| 시간 절약 | 문서 작성에 반나절을 쓰지만 그 후 디버깅 시간이 줄어듦 |
| 품질 | LLM이 만든 코드의 일관성이 크게 향상 |
| 비용 대비 효율 | 매우 높음 — LLM 호출이 줄어듦 |
| 추천 여부 | 강하게 추천 |

## 이런 사람에게 추천

- **추천 대상**: 사이드 프로젝트지만 1년 이상 유지할 생각이 있는 사람
- **비추천 대상**: 하루 만에 만들고 끝낼 일회성 사이트를 원하는 사람
